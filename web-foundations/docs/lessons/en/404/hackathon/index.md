**Duration:** 2 hours (with 10 days for refinement)

This hackathon challenges you to build a creative "Page Not Found" experience with a space/Mars theme. You'll work in teams to design, develop, and deploy a custom 404 error page that stands out through innovation, design quality, and clean code.

A custom 404 error page that:

- **Entertains** users who land on a non-existent page

- **Demonstrates** your HTML/CSS/JavaScript skills
- **Works** responsively across devices

## Introduction

Welcome to the **404s to Mars** hackathon challenge! In this 2-hour sprint, your team students will create an innovative web page ‚Äì specifically, a custom 404 error page ‚Äì inspired by the theme "404s to Mars".

The goal is to design a fun and creative "Page Not Found" experience (think lost in space on Mars!) while demonstrating your coding and design skills under time pressure. Although the contest phase lasts only 2 hours, remember you'll have about 10 days afterward to refine the project for course submission, so use this hackathon to build a solid prototype.

This brief will guide you through how to organize your team, build the project, and what criteria the judges (and your instructor) will value in your final product.

## Evaluation Criteria üìã

Keep these judging criteria in mind as you plan and build your project ‚Äì they indicate what will be valued in your hackathon submission:

### 1. Innovation & Originality

Show off your creativity! Judges look for projects that push boundaries or take a novel approach to the problem. This means your 404 page idea should stand out ‚Äì perhaps an unexpected concept, a clever theme, or a unique user experience that surprises and delights.

**What to consider:**

- Don't be afraid to think outside the box
- An original solution or design can set your project apart
- Aim to do something that hasn't been seen before, or put a new twist on an old idea
- Consider interactive elements, storytelling, or gamification

### 2. Quality of Design (UI/UX)

The visual design and user experience of your page are crucial. A polished, well-thought-out design will make a strong impression. Ensure your 404 page is not only visually appealing but also easy to use and understand.

**Good design involves:**

- Clear layout and visual hierarchy
- Readable typography
- Appealing graphics and color scheme
- Intuitive navigation (e.g., clear call-to-action to return home)
- Responsive design that works on mobile and desktop

> **Note:** Hackathon judges often heavily reward good usability and interface design ‚Äì some events weight design and usability at 50% of the judging criteria.

### 3. Quality of Code (Technical Execution)

Clean, efficient, and well-structured code is the backbone of a quality project. Even in a short hackathon, strive for solid technical execution.

**Key technical aspects:**

- Writing clear and maintainable HTML/CSS/JS
- Using proper folder structure
- Avoiding sloppy hacks or quick-and-dirty solutions
- Handling edge cases
- Ensuring cross-browser and cross-device compatibility
- Using version control (GitHub) effectively with logical commits

By focusing on creativity, design polish, and code craftsmanship, you'll address the key points the judges are looking for.

### Typing Animation Effect

```javascript
function typeWriter(element, text, speed = 50) {
	let i = 0;
	element.textContent = '';

	function type() {
		if (i < text.length) {
			element.textContent += text.charAt(i);
			i++;
			setTimeout(type, speed);
		}
	}

	type();
}

// Usage:
const title = document.querySelector('.error-title');
```

### Particle Background (Simple Version)

```javascript
class Star {
	constructor(canvas) {
		this.canvas = canvas;
		this.x = Math.random() * canvas.width;
		this.y = Math.random() * canvas.height;
		this.size = Math.random() * 2;
		this.speedY = Math.random() * 0.5;
	}

	update() {
		this.y += this.speedY;
		if (this.y > this.canvas.height) {
			this.y = 0;
			this.x = Math.random() * this.canvas.width;
		}
	}

	draw(ctx) {
		ctx.fillStyle = 'white';
		ctx.beginPath();
		ctx.arc(this.x, this.y, this.size, 0, Math.PI * 2);
		ctx.fill();
	}
}

// Initialize canvas and stars
const canvas = document.getElementById('starfield');
const ctx = canvas.getContext('2d');
canvas.width = window.innerWidth;
canvas.height = window.innerHeight;

const stars = Array(100)
	.fill()
	.map(() => new Star(canvas));

function animate() {
	ctx.fillStyle = 'rgba(10, 14, 39, 0.1)';
	ctx.fillRect(0, 0, canvas.width, canvas.height);

	stars.forEach((star) => {
		star.update();
		star.draw(ctx);
	});

	requestAnimationFrame(animate);
}

animate();
```

## Pro Tips & Best Practices ‚≠ê

### Time Management

1. **Stick to the timeline** - Set phone alarms for each phase
2. **Timebox decisions** - Don't spend 20 minutes choosing fonts
3. **MVP first** - Get basics working, then add polish
4. **Cut features ruthlessly** - Better to have one good thing than three half-done things

### Collaboration

1. **Communicate constantly** - Use voice chat or sit together
2. **Commit often** - Small, frequent commits prevent conflicts
3. **Pull before pushing** - Always sync before your push
4. **Use branches (if time)** - Main features on branches, merge when done

### Design

1. **Limit your palette** - 3-4 colors maximum
2. **Consistent spacing** - Use a spacing scale (8px, 16px, 24px, etc.)
3. **One strong focal point** - Usually the "404" error code
4. **White space is your friend** - Don't cram everything together

### Code

1. **Mobile-first CSS** - Base styles for mobile, media queries for desktop
2. **Use CSS variables** - Makes theming and adjustments quick
3. **Comment your code** - Future you (in 10 days) will thank present you
4. **Test frequently** - Don't wait until the end to open the page

### Avoiding Common Pitfalls

‚ùå **Don't:**

- Spend too long on perfect pixel alignment
- Add complex JavaScript if plain CSS works
- Use huge image files (optimize first!)
- Try to learn a new framework during the hackathon
- Forget to test on mobile

‚úÖ **Do:**

- Keep it simple and polished
- Use established patterns and libraries
- Compress images before adding to repo
- Stick with technologies you know
- Check responsive design early and often

## FAQ ‚ùì

### Q: Can we use external libraries?

**A:** Yes! Using CSS frameworks (Tailwind, Bootstrap) or animation libraries is encouraged if it helps you build faster. Just ensure you understand what you're using and can explain it.

### Q: What if we don't finish in 2 hours?

**A:** That's expected! The hackathon is about building a solid prototype. You have 10 days afterward to refine and complete the project for final submission.

### Q: How important is the README?

**A:** Very! The README is your project's first impression. It shows you can document your work professionally, which is valued in the industry.

### Q: Can we change our idea halfway through?

**A:** Only if absolutely necessary. Major pivot after 60 minutes will likely mean you won't finish. Better to simplify your current idea than start over.

### Q: What if team members have different skill levels?

**A:** Perfect opportunity to learn from each other! Pair up stronger members with those learning. Everyone contributes value ‚Äì design, content, testing all matter.

### Q: Should we prioritize design or functionality?

**A:** For a 404 page, design and UX are paramount since functionality is minimal (it's mostly static content). However, clean code is still important for evaluation.

### Q: Can we use AI tools like ChatGPT or GitHub Copilot?

**A:** Check with your instructor. Generally, using AI for suggestions is fine, but you must understand and be able to explain all code you submit.

### Q: What's the minimum viable product?

**A:** At minimum:

- A working HTML page named `404.html`
- Mars/space theme evident
- Clear error message
- Navigation back to home
- Mobile-responsive
- Deployed on GitHub Pages

Anything beyond that is enhancement!
